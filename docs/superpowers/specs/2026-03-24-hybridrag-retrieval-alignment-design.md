# HybridRAG Retrieval Alignment — Design Spec
Date: 2026-03-24

## Problem Statement

The hybrid RAG system has two broken retrieval paths:

1. **Vector search (arxiv_chunks)**: The 34,379 chunk points in Qdrant store pre-computed embeddings from the HybridRAG-Bench dataset, which were generated by an undocumented third-party model. At query time, the pipeline embeds queries with SPECTER2. Cosine similarity between stored dataset vectors and SPECTER2 vectors for the same text = **0.023** (near-zero). This makes `arxiv_chunks` effectively unsearchable — the right chunks exist but cannot be found.

2. **Graph retrieval (GraphRetriever)**: The `CSEntityExtractor` extracts CS/AI technical terms (model names, frameworks, datasets), but the knowledge graph stores general domain entities — Concept, Person, Organization, Method, Software — as ALL-CAPS display names. The exact-name lookup in Neo4j (`CONTAINS toLower($name)`) rarely matches because the extractor's vocabulary does not match the graph's entity naming. Named entities like "McCaskill", "Friston", "FitzGerald" are absent from the extractor's dictionaries entirely.

3. **arxiv_nodes**: Also stores dataset pre-computed embeddings (same mismatch). Once fixed, this collection enables semantic entity lookup to replace the broken exact-name approach.

**Evidence:**
- Direct SPECTER2-to-SPECTER2 similarity for McCaskill question vs. relevant chunk: 0.73–0.80
- Qdrant query with SPECTER2 against stored dataset vectors: top score = 0.12 (wrong papers returned)
- `import_hf_kg.py` confirms dataset intended Neo4j native vector indexes, not Qdrant; embedding model is undocumented

---

## Approach

Re-embed `arxiv_chunks` and `arxiv_nodes` with SPECTER2 at ingestion time (same model already used for `arxiv_papers` and queries). Replace the exact-name entity lookup with semantic vector search against the re-embedded `arxiv_nodes` collection.

`arxiv_papers` is already correct — no change needed.

## Component Changes

### 1. `src/pipelines/local_ingestion_pipeline.py`

#### `_ingest_nodes(domain)`

**Current behaviour**: the outer loop appends both a `NodeRecord` to `batch` and a `PointStruct(vector=rec.embedding, ...)` to `qdrant_points` using the dataset pre-computed vector. `_flush()` upserts `qdrant_points` then clears both lists.

**Change**:
- Remove the `qdrant_points` list variable from the outer scope entirely.
- Remove the `qdrant_points.append(...)` call from the outer loop. The outer loop appends only `NodeRecord` objects to `batch`.
- Inside `_flush()`, after the Neo4j write:
  1. Build both a `display_names` list and a parallel `embeddable_recs` list from `batch`, skipping records where `display_name` is empty or `None`. Both lists must be the same length and in the same order.
  2. Call `embed_texts_with_model(display_names, EMBEDDING_MODEL, batch_size=64)` to get SPECTER2 vectors.
  3. Build a local `qdrant_points` list (scoped to `_flush()`) by zipping `embeddable_recs` (not the full `batch`) with their embeddings.
  4. Upsert `qdrant_points` to `arxiv_nodes`.
  5. Clear only `batch` (the `qdrant_points` list is local to `_flush()` and needs no explicit clear).

**Edge case**: if `rec.display_name` is empty or `None`, skip that record for Qdrant (do not embed; do not upsert). It is still written to Neo4j as part of the Neo4j batch.

**Progress counter**: the existing `total` counter (e.g., `total += len(batch)`) counts records written to Neo4j and must NOT be changed to `len(embeddable_recs)`. Records skipped for Qdrant are still Neo4j writes and should be counted.

#### `_ingest_chunks(domain, include_src_names, progress)`

**Current behaviour**: the outer loop appends a `PointStruct(vector=rec.embedding, ...)` to `points`. `_flush()` constructs `neo4j_batch` by reading `p.payload.get("src_id")`, `p.payload.get("dst_id")`, etc. from those `PointStruct` objects, upserts to Qdrant, then writes `HAS_CHUNK` edges to Neo4j.

**Change**:
- Rename the outer accumulator from `points: list[PointStruct]` to `recs: list[ChunkRecord]`.
- In the outer loop, append the raw `ChunkRecord` (not a `PointStruct`) to `recs`.
- Inside `_flush()`:
  1. Build `neo4j_batch` from `ChunkRecord` attributes — `rec.src_id`, `rec.dst_id`, `rec.edge_id`, `rec.domain`, `str(rec.qdrant_id)` — instead of from `p.payload` dict lookups. This preserves the existing error-recovery and HAS_CHUNK write logic exactly.
  2. Embed paragraphs: call `embed_texts_with_model([r.paragraph for r in recs], EMBEDDING_MODEL, batch_size=64)`.
  3. Build a local `points: list[PointStruct]` by zipping `recs` with their embeddings, carrying the same payload dict as before.
  4. Upsert `points` to `arxiv_chunks`.
  5. Clear only `recs`.

The existing `qdrant_ok` flag, `progress.log_batch_failure` call, and HAS_CHUNK Neo4j write remain structurally unchanged. Only the source of the vector (embedding at flush time vs. pre-computed at loop time) and the type of the accumulator change.

**Flush threshold**: `_ingest_chunks` uses `QDRANT_BATCH` (256) as its flush threshold (not `self.neo4j_batch_size`). This threshold applies to the renamed `recs` accumulator exactly as it did to the old `points` accumulator — no change to the flush condition.

#### `_ingest_paper_chunks(domain, progress)` — no change (already embeds with SPECTER2)

**Re-ingestion requirement**: run with `--reset` to wipe all Qdrant collections and Neo4j data before starting. This is all-or-nothing: once reset completes, partial failure during ingestion leaves the database in an incomplete state. The progress file tracks per-domain phase completion so a resumed run will continue from where it left off within a domain, but domains already wiped before failure are gone. This is an accepted operational risk for a development environment.

### 2. `src/graph/cs_knowledge_graph.py`

Add one method. The Cypher uses an undirected relationship match (`-[r:HAS_CHUNK]-()`) to return chunk refs regardless of whether the matched node is the `src` or `dst` of the HAS_CHUNK edge:

```python
def get_chunk_refs_by_node_ids(self, node_ids: list[int], limit: int = 20) -> list[str]:
    """
    Return qdrant_id strings from HAS_CHUNK edges connected to _Embeddable
    nodes whose node_id is in the given list. Matches both src and dst
    sides of the HAS_CHUNK relationship.
    """
    query = """
    MATCH (n:_Embeddable)-[r:HAS_CHUNK]-()
    WHERE n.node_id IN $node_ids
    RETURN DISTINCT r.qdrant_id AS qdrant_id
    LIMIT $limit
    """
    rows = self.client.execute_read(query, {"node_ids": node_ids, "limit": limit})
    return [r["qdrant_id"] for r in rows if r.get("qdrant_id")]
```

The existing `get_chunk_refs_for_entity(display_name)` method is retained but no longer called by `GraphRetriever.retrieve()`. `retrieve_by_entity()` still calls it; that method is not invoked by the pipeline and is excluded from this fix scope (separate concern).

### 3. `src/retrieval/graph_retriever.py`

#### `GraphRetriever.__init__`
- Add `qdrant_url: str = "http://localhost:6333"` parameter.
- Instantiate `self._qdrant = QdrantClient(url=qdrant_url, timeout=30)`.
- Add module-level constant `NODES_PER_ENTITY = 3` and `EMBEDDING_MODEL = "allenai/specter2_base"`.
- Import `embed_texts_with_model` from `src.utils` and `QdrantClient` from `qdrant_client`.

#### `GraphRetriever.retrieve(query, top_k, entity_types)`

Replace the `self.kg.get_chunk_refs_for_entity(name, ...)` call with:

```
for each entity_name in entity_names:
    1. name_emb = embed_texts_with_model([name], EMBEDDING_MODEL, batch_size=1)[0]
    2. results = self._qdrant.query_points("arxiv_nodes", query=name_emb,
                                           with_payload=True, limit=NODES_PER_ENTITY)
    3. node_ids = [r.payload["node_id"] for r in results.points
                   if r.payload and r.payload.get("node_id") is not None]
    4. self._last_trace["qdrant_ids_per_entity"][name] = len(node_ids)
    5. if node_ids:
           ids = self.kg.get_chunk_refs_by_node_ids(node_ids, limit=top_k * 2)
           all_qdrant_ids.extend(ids)
```

Note on trace field: `_last_trace["qdrant_ids_per_entity"][name]` now records matched **node count** (from arxiv_nodes search), not qdrant_id count (from Neo4j traversal). The semantics change from "how many chunk IDs did this entity yield" to "how many graph nodes matched this entity name". This is a deliberate change; callers reading the trace for debugging should be aware. Renaming the field to `nodes_per_entity` is explicitly deferred to a separate cleanup task — do not rename it in this change.

#### `retrieve_by_entity(entity_name, top_k)` — excluded from this fix

This method is not called by `HybridRAGBenchPipeline` or `HybridRetriever`. It retains the existing `get_chunk_refs_for_entity` call. Fixing it is a separate task.

#### `get_graph_retriever()` singleton — excluded from this fix

`graph_retriever.py` exposes a module-level `get_graph_retriever()` function that caches a `GraphRetriever` instance. `HybridRAGBenchPipeline` constructs `GraphRetriever` directly and never calls this singleton, so the singleton is not in the call path affected by this fix. The singleton signature is not updated — any caller using it will get the default `qdrant_url`. This is an accepted limitation of this change scope.

#### `HybridRAGBenchPipeline` (`src/pipelines/hybridrag_pipeline.py`)

Pass `chunk_retriever=self.chunk_retriever` to `GraphRetriever` (already done). No additional change needed — `qdrant_url` defaults to `http://localhost:6333` which matches the rest of the system.

## Data Flow After Fix

```
Query
  │
  ├─► SPECTER2 embed ──────────────────────────────────────────────────────────┐
  │                                                                             │
  ├─► ChunkRetriever.search(arxiv_chunks)  ← SPECTER2 vectors (re-embedded) ◄──┤
  │                                                                             │
  ├─► PaperRetriever.search(arxiv_papers) ← SPECTER2 vectors (already ok)  ◄──┤
  │                                                                             │
  └─► GraphRetriever.retrieve()                                                 │
        │                                                                       │
        ├─► CSEntityExtractor → entity names                                    │
        ├─► SPECTER2 embed each name ◄───────────────────────────────────────────┘
        ├─► arxiv_nodes Qdrant search → node_ids  ← SPECTER2 vectors (re-embedded)
        ├─► Neo4j: get_chunk_refs_by_node_ids → qdrant_ids (src or dst match)
        └─► ChunkRetriever.fetch_by_ids → RetrievedContext list

All three paths → HybridRetriever RRF fusion → Reranker → LLM
```

## Verification Plan

After re-ingestion, run a diagnostic script to confirm alignment:

**Step 1 — Vector search check**
- Embed: "What concept was developed by McCaskill and is the basis for the software Contrafold?"
- Search arxiv_chunks top-10
- Pass condition: paper `1912.13190v3` appears in top-5, score > 0.5

**Step 2 — Entity semantic lookup check**
- Embed: "PARTITION FUNCTION"
- Search arxiv_nodes top-5
- Pass condition: at least one result with `entity_type=Concept` and display_name containing relevant terms

**Step 3 — Graph retrieval end-to-end check**
- Run `GraphRetriever.retrieve("What concept was developed by McCaskill...")`
- Pass condition: `fetched_count > 0`, contexts from paper `1912.13190v3`

**Step 4 — Full pipeline spot check**
- Run `pipeline.query()` on 3 previously-failing questions (McCaskill, Friston, Genesis)
- Pass condition: `answer_type = "answer"` (not refusal), reranker top score > 0
